## 🎯 ЧТО ТАКОЕ FIREWALL?

**Firewall (Брандмауэр)** — это система безопасности, которая контролирует сетевой трафик между компьютерами/серверами.

**Простая аналогия:**
```
Firewall = Охранник на входе в здание

Охранник проверяет:
- Кто хочет войти? (IP адрес)
- Откуда пришёл? (Источник)
- Куда идёт? (Назначение)
- Что несёт? (Тип трафика)
- Есть ли пропуск? (Правила доступа)

Если всё ОК → пропускает
Если что-то подозрительно → блокирует
```

## 🏗️ АРХИТЕКТУРА СЕТИ В IaaS

### Без Firewall (небезопасно):

```
Интернет
   │
   ├─→ VM Tenant A (10.10.1.10) ← открыта для всех
   ├─→ VM Tenant B (10.10.1.20) ← открыта для всех
   └─→ VM Tenant C (10.10.1.30) ← открыта для всех

Проблемы:
❌ Любой из интернета может подключиться к любой VM
❌ Tenant A может атаковать Tenant B
❌ Нет контроля портов (SSH, HTTP, HTTPS)
❌ Нет защиты от DDoS атак
```

### С Firewall (безопасно):

```
Интернет
   │
   ▼
┌─────────────────────────────────┐
│      Firewall (Правила)        │
│  ────────────────────────────  │
│  ✅ Разрешено:                 │
│     - 80 (HTTP) → VM web-01    │
│     - 443 (HTTPS) → VM web-01   │
│     - 22 (SSH) → только с VPN  │
│  ❌ Запрещено:                 │
│     - Все остальные порты       │
│     - Прямой доступ между VM   │
└─────────────┬───────────────────┘
              │
              ├─→ VM Tenant A (10.10.1.10)
              ├─→ VM Tenant B (10.10.1.20)
              └─→ VM Tenant C (10.10.1.30)

Преимущества:
✅ Контроль доступа
✅ Изоляция между тенантами
✅ Защита от атак
✅ Логирование попыток доступа
```

## 🎯 РОЛЬ FIREWALL В IaaS ПЛАТФОРМЕ

### 1. **Изоляция между тенантами**

**Проблема:** В мультитенантной среде нужно изолировать сетевой трафик между разными клиентами.

**Решение Firewall:**

```
Tenant A:
  VM web-01 (10.10.1.10) → может общаться только с:
    - VM db-01 (10.10.1.11) ← свой тенант
    - Интернет (для веб-трафика)
    ❌ НЕ может общаться с Tenant B

Tenant B:
  VM api-01 (10.10.2.10) → может общаться только с:
    - VM cache-01 (10.10.2.11) ← свой тенант
    - Интернет
    ❌ НЕ может общаться с Tenant A

Firewall правила:
  ALLOW: 10.10.1.0/24 → 10.10.1.0/24  (Tenant A внутренняя сеть)
  ALLOW: 10.10.2.0/24 → 10.10.2.0/24  (Tenant B внутренняя сеть)
  DENY:  10.10.1.0/24 → 10.10.2.0/24  (Tenant A → Tenant B)
  DENY:  10.10.2.0/24 → 10.10.1.0/24  (Tenant B → Tenant A)
```

**Результат:** Даже если злоумышленник получит доступ к VM Tenant A, он не сможет атаковать Tenant B.

---

### 2. **Контроль доступа из интернета**

**Проблема:** Не все порты должны быть открыты для всех.

**Решение Firewall:**

```
Сценарий: Клиент создал веб-сервер

Без Firewall:
  ❌ Все порты открыты (0-65535)
  ❌ Злоумышленник может попробовать подключиться к:
     - 22 (SSH) → попытка взлома
     - 3306 (MySQL) → попытка взлома БД
     - 5432 (PostgreSQL) → попытка взлома БД
     - 6379 (Redis) → попытка взлома кэша

С Firewall:
  ✅ Открыты только нужные порты:
     - 80 (HTTP) → для веб-сайта
     - 443 (HTTPS) → для защищённого веб-сайта
  ❌ Все остальные порты закрыты
  ✅ SSH (22) доступен только через VPN или с белого списка IP
```

**Пример правил:**

```
VM: web-server-01 (10.10.1.10)

Firewall Rules:
  ✅ ALLOW: 0.0.0.0/0 → 10.10.1.10:80   (HTTP из интернета)
  ✅ ALLOW: 0.0.0.0/0 → 10.10.1.10:443  (HTTPS из интернета)
  ✅ ALLOW: 203.0.113.0/24 → 10.10.1.10:22  (SSH только с офиса клиента)
  ❌ DENY:  все остальное
```

---

### 3. **Защита от DDoS атак**

**Проблема:** Злоумышленник может отправить миллионы запросов на сервер, чтобы его перегрузить.

**Решение Firewall:**

```
Firewall может:
  - Ограничить количество соединений с одного IP
  - Блокировать подозрительные паттерны трафика
  - Распределять нагрузку между серверами
  - Автоматически блокировать атакующие IP

Пример:
  Если с IP 192.0.2.100 приходит >100 запросов/сек:
    → Firewall автоматически блокирует этот IP на 1 час
```

---

### 4. **Логирование и аудит**

**Проблема:** Нужно знать, кто и когда пытался получить доступ.

**Решение Firewall:**

```
Firewall логирует:
  - Все попытки подключения
  - Заблокированные запросы
  - Источник и назначение
  - Время и тип атаки

Пример лога:
  2025-03-04 12:34:56 | BLOCKED | 192.0.2.100 → 10.10.1.10:3306 | MySQL port scan
  2025-03-04 12:35:01 | BLOCKED | 192.0.2.100 → 10.10.1.10:22   | SSH brute force
  2025-03-04 12:35:10 | ALLOWED | 203.0.113.50 → 10.10.1.10:443 | HTTPS request
```

## 🔐 ТИПЫ FIREWALL В IaaS

### 1. **Network Firewall (Сетевой)**

**Где:** На уровне сети провайдера (перед всеми VM)

**Что делает:**
- Контролирует трафик между интернетом и VM
- Контролирует трафик между VM разных тенантов
- Работает на уровне IP адресов и портов

**Пример:**
```
┌─────────────────────────────────┐
│   Network Firewall (Провайдер)  │
│  ─────────────────────────────  │
│  Правила для всех тенантов      │
└─────────────┬───────────────────┘
              │
              ├─→ Tenant A VMs
              ├─→ Tenant B VMs
              └─→ Tenant C VMs
```

---

### 2. **VM-level Firewall (На уровне VM)**

**Где:** Внутри каждой виртуальной машины

**Что делает:**
- Контролирует трафик конкретной VM
- Настраивается клиентом самостоятельно
- Работает через iptables (Linux) или Windows Firewall

**Пример:**
```
VM web-server-01:
  $ sudo ufw allow 80/tcp    # Разрешить HTTP
  $ sudo ufw allow 443/tcp   # Разрешить HTTPS
  $ sudo ufw deny 22/tcp     # Запретить SSH из интернета
  $ sudo ufw enable          # Включить firewall
```

---

### 3. **Application Firewall (WAF)**

**Где:** Перед веб-приложениями

**Что делает:**
- Анализирует HTTP/HTTPS запросы
- Блокирует SQL-инъекции, XSS атаки
- Защищает от ботов и сканеров

**Пример:**
```
Запрос: GET /index.php?id=1' OR '1'='1
         ↓
WAF анализирует:
  - Обнаружена SQL-инъекция (' OR '1'='1)
  - Блокирует запрос
  - Логирует попытку атаки
```

### Как это работает в реальных провайдерах:

#### AWS (Security Groups):

```
Клиент создаёт Security Group (группу правил firewall):

Security Group: "web-server-sg"
  ✅ Inbound Rules:
     - Type: HTTP, Port: 80, Source: 0.0.0.0/0
     - Type: HTTPS, Port: 443, Source: 0.0.0.0/0
     - Type: SSH, Port: 22, Source: 203.0.113.0/24  (только офис)
  ❌ Outbound Rules:
     - Type: All traffic, Destination: 0.0.0.0/0

VM привязывается к Security Group:
  VM web-01 → Security Group: "web-server-sg"
  
Результат:
  ✅ VM получает только разрешённый трафик
  ❌ Все остальное блокируется
```

#### DigitalOcean (Cloud Firewalls):

```
Клиент создаёт Firewall правила через веб-интерфейс:

Firewall: "production-fw"
  ✅ Inbound:
     - HTTP (80) from anywhere
     - HTTPS (443) from anywhere
     - SSH (22) from office IP only
  ❌ Outbound:
     - All traffic allowed

Применяется к:
  - Droplet web-01
  - Droplet db-01
```

## 🎯 КАК FIREWALL СВЯЗАН С МУЛЬТИТЕНАНТНОСТЬЮ

### Проблема без Firewall:

```
Tenant A создал VM:
  - web-01 (10.10.1.10)
  - db-01 (10.10.1.11)

Tenant B создал VM:
  - api-01 (10.10.2.10)

БЕЗ Firewall:
  ❌ Tenant A может подключиться к api-01 Tenant B
  ❌ Tenant B может подключиться к db-01 Tenant A
  ❌ Нет сетевой изоляции
```

### Решение с Firewall:

```
Tenant A:
  Firewall Rules:
    ✅ ALLOW: 10.10.1.10 ↔ 10.10.1.11  (web-01 ↔ db-01)
    ✅ ALLOW: 0.0.0.0/0 → 10.10.1.10:80,443  (интернет → веб)
    ❌ DENY:  всё остальное

Tenant B:
  Firewall Rules:
    ✅ ALLOW: 10.10.2.10 → интернет  (api-01 → внешние API)
    ✅ ALLOW: интернет → 10.10.2.10:443  (HTTPS)
    ❌ DENY:  всё остальное

Результат:
  ✅ Tenant A НЕ может подключиться к Tenant B
  ✅ Tenant B НЕ может подключиться к Tenant A
  ✅ Каждый тенант изолирован на сетевом уровне
```

```
  ✅ ALLOW: 10.10.1.10 → 0.0.0.0/0  (всё разрешено для исходящего)

Результат:
  ✅ Сайт доступен из интернета
  ❌ SSH, MySQL, Redis закрыты
  ✅ Защита от сканирования портов
```

---

### Пример 2: База данных

**Задача:** Клиент создал MySQL сервер, доступ должен быть только из внутренней сети.

**Firewall правила:**

```
VM: db-01 (10.10.1.11)

Inbound Rules:
  ✅ ALLOW: 10.10.1.10 → 10.10.1.11:3306  (только от web-01)
  ✅ ALLOW: 10.10.1.12 → 10.10.1.11:3306  (только от api-01)
  ❌ DENY:  всё остальное (включая интернет)

Outbound Rules:
  ✅ ALLOW: 10.10.1.11 → 0.0.0.0/0  (для обновлений)

Результат:
  ✅ БД доступна только из внутренней сети тенанта
  ❌ БД НЕ доступна из интернета
  ✅ Защита от SQL-инъекций извне
```

---

### Пример 3: API сервер

**Задача:** Клиент создал API сервер, доступ только по HTTPS с авторизацией.

**Firewall правила:**

```
VM: api-01 (10.10.2.10)

Inbound Rules:
  ✅ ALLOW: 0.0.0.0/0 → 10.10.2.10:443  (HTTPS из интернета)
  ❌ DENY:  0.0.0.0/0 → 10.10.2.10:80   (HTTP запрещён)
  ❌ DENY:  всё остальное

Outbound Rules:
  ✅ ALLOW: 10.10.2.10 → 0.0.0.0/0  (API может вызывать внешние сервисы)

Результат:
  ✅ API доступен только по HTTPS (зашифровано)
  ❌ HTTP закрыт (небезопасно)
  ✅ Защита от перехвата трафика
```

### Вариант 2: Реальная интеграция (для production)

**Использовать реальный firewall API (например, iptables через Proxmox):**

```javascript
// backend/src/services/firewallService.js

const { exec } = require('child_process');

async function applyFirewallRule(vm, rule) {
  // Получаем реальный IP VM из Proxmox
  const vmIP = await getVMIPFromProxmox(vm.id);
  
  if (rule.action === 'allow') {
    // iptables -A INPUT -s <source> -d <vmIP> -p <protocol> --dport <port> -j ACCEPT
    await exec(`iptables -A INPUT -s ${rule.source} -d ${vmIP} -p ${rule.protocol} --dport ${rule.port} -j ACCEPT`);
  } else {
    // iptables -A INPUT -s <source> -d <vmIP> -p <protocol> --dport <port> -j DROP
    await exec(`iptables -A INPUT -s ${rule.source} -d ${vmIP} -p ${rule.protocol} --dport ${rule.port} -j DROP`);
  }
}
```

**⚠️ Внимание:** Это требует root-доступа к гипервизору и может быть опасно.

## 📋 ТАБЛИЦА: Сравнение с/без Firewall

| Аспект | Без Firewall | С Firewall |
|--------|--------------|------------|
| **Доступ из интернета** | Все порты открыты | Только разрешённые порты |
| **Изоляция тенантов** | ❌ Нет (только на уровне данных) | ✅ Да (сетевая изоляция) |
| **Защита от атак** | ❌ Нет | ✅ Да (блокировка подозрительного трафика) |
| **Контроль портов** | ❌ Нет | ✅ Да (клиент настраивает правила) |
| **Логирование** | ❌ Нет | ✅ Да (все попытки доступа) |
| **DDoS защита** | ❌ Нет | ✅ Да (ограничение соединений) |
| **Безопасность БД** | ❌ БД может быть доступна из интернета | ✅ БД только из внутренней сети |

## 🎓 ВЫВОДЫ ДЛЯ ПРОДАКТ-МЕНЕДЖЕРА

### ✅ Зачем нужен Firewall в IaaS:

1. **Безопасность клиентов**
   - Защита от атак из интернета
   - Контроль доступа к портам
   - Изоляция между тенантами

2. **Соответствие требованиям**
   - Многие клиенты требуют сетевую изоляцию
   - Соответствие стандартам безопасности (ISO 27001, SOC 2)
   - Защита персональных данных (GDPR)

3. **Конкурентное преимущество**
   - "Мы обеспечиваем сетевую изоляцию" — важный аргумент продаж
   - Клиенты готовы платить больше за безопасность

4. **Снижение рисков**
   - Меньше инцидентов безопасности
   - Меньше компенсаций клиентам
   - Защита репутации провайдера

---

### ⚠️ Что НЕ реализовано в нашем проекте:

1. ❌ **Network Firewall** — нет правил на уровне сети
2. ❌ **VM-level Firewall** — нет управления правилами для VM
3. ❌ **Сетевая изоляция** — тенанты могут "видеть" друг друга в сети
4. ❌ **Логирование трафика** — нет логов попыток доступа

**Почему:** Это MVP/прототип, фокус на демонстрации мультитенантности данных, а не сетевой изоляции.
